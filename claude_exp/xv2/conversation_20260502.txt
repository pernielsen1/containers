CLAUDE: Concurrency discussion:

Option A — asyncio:
  - Non-blocking crypto calls expressed naturally with await
  - No locks needed on shared state (single event loop)
  - Command server runs as coroutine in same loop via aiohttp
  - Downside: async/await is infectious; everything must be async

Option B — threading (matches auth_test style):
  - One thread per upstream connection
  - Flask handles command server in daemon thread
  - pyiso8583 is synchronous — fits without wrappers
  - Need threading.Lock on pending dict and stats counters
  - Downside: explicit locking everywhere shared state is touched

Recommendation: asyncio — the router's core constraint ("don't block upstream
while waiting for crypto") writes itself naturally with await.

CLAUDE: Portability analysis:

asyncio → C++:
  asyncio's async/await is Python-specific syntax. C++ equivalent (C++20
  coroutines or Boost.Asio coroutines) has completely different syntax and
  semantics — would require a conceptual rewrite, not a port. Code doesn't
  serve as a useful blueprint.

threading → C++:
  Maps almost 1:1:
    threading.Thread        →  std::thread
    threading.Lock          →  std::mutex
    blocking socket read    →  recv()
    queue.Queue             →  std::queue + std::condition_variable

  A Python threading implementation is essentially executable pseudocode
  for the C++ version.

REVISED RECOMMENDATION: Threading.
  If a C++ port is a realistic future (even a maybe), the threading model gives
  a working Python reference implementation that doubles as a design document
  for C++. asyncio gives nothing portable. The cost — explicit locks on the
  pending-requests dict and stats counters — is manageable and good practice
  for the C++ version anyway.
